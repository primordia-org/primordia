# Unify exe.dev and passkey login into a single account

Previously, logging in via exe.dev and registering a passkey created two separate, unrelated user accounts. Users who signed in with exe.dev had no way to access their account via a passkey, and vice versa.

## What changed

- After a successful exe.dev login, users who have **no passkeys yet** are redirected to a new `/register-passkey` interstitial page that prompts them to register a passkey for their account. They can skip if they prefer.
- The passkey registration flow now supports two modes:
  - **Logged-in user (new):** if a valid session exists when `/api/auth/passkey/register/start` is called, the challenge is bound to the existing user's ID. The finish endpoint then adds the new passkey to that account instead of creating a fresh user. This allows exe.dev accounts (whose usernames are email addresses and wouldn't pass the alphanumeric-only username rule) to register passkeys without hitting validation errors.
  - **New user (unchanged):** the original flow still works when no session is present.
- The new `/register-passkey` page is a lightweight server + client component pair. The server component redirects to `/login` if there is no session, ensuring the endpoint is always authenticated.

## Why

Passkey authentication and exe.dev SSO were operating as completely separate identity silos. This change ties them together so that after one-time passkey enrolment following an exe.dev login, the user can sign in via either method and always land in the same account.
