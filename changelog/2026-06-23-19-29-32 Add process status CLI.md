# Add process status CLI

Implemented a small process manager library and `bun run process status` CLI for inspecting Primordia worktrees without asking the reverse proxy for process state. Git worktree/config reads are isolated in `lib/git-runtime.ts` so process inspection stays separate from repository metadata lookup, and table rendering lives in `lib/cli-utils.ts` for reuse.

The new status command lists each registered non-bare git worktree, its assigned branch port, any active Next.js listener on that port, the server process state, inferred environment, server PID(s), descendant child-process counts, and active agent worker PID(s). Agent labels are derived generically from the worker `.ts` filename instead of a hard-coded harness list. It skips the bare source repository entry from `git worktree list --porcelain`, omits unneeded git HEAD values, and sorts worktrees by assigned port. The default output renders a dedicated reverse-proxy table followed by a compact worktree table (`Worktree`, `Port`, `State`, `Env`, `PID`, `Children`, `Agents`) while `--json` returns full details including `reverseProxy`, worktree paths, `childPids`, and a per-worktree `servers` array, where an empty array means no active listener and multiple entries indicate an abnormal multi-listener situation. The CLI can also start, stop, or restart a worktree server directly with `bun run process <worktreename> start [--dev|--prod]`, `stop`, and `restart [--dev|--prod]`, using the worktree's assigned branch port without involving the reverse proxy; all command actions support `--json`. Started servers append stdout/stderr to `.primordia-next-server.log` in the worktree, and `bun run process <worktreename> logs [--follow] [--json]` reads or follows that log via the shared process-manager log iterator. The evolve session Web Preview card now streams that log with the shared `SuspenseLogFile` server component and restarts preview servers through process-manager server actions instead of the reverse proxy's preview log/restart endpoints; the now-unused `/_proxy/preview/:id/logs` route has been removed. The reverse proxy delegates lazy preview dev-server spawning/stopping to the same process-manager layer. The installed reverse proxy resolves the process-manager helper from the relevant worktree so copied-proxy deployments do not need separate helper-file copies. Continuous refresh is intentionally left to standard tools like Linux `watch`.
