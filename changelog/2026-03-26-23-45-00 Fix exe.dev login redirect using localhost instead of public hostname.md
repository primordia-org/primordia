# Fix exe.dev login redirect using localhost instead of public hostname

## What changed

Added a `getPublicOrigin()` helper in `app/api/auth/exe-dev/route.ts` and
used it as the base URL for both `NextResponse.redirect()` calls in that route,
replacing the previous use of `req.url`.

## Why

When Primordia runs behind exe.dev's reverse proxy (e.g. at
`https://primordia.exe.xyz:3000`), Next.js only sees the internal request URL,
which has `localhost` as the hostname. The proxy terminates TLS and forwards
traffic to `localhost:PORT` on the server, so `req.url` and `req.nextUrl` both
contain `localhost` rather than the public-facing hostname.

Both redirect targets in the exe.dev auth route were constructed as
`new URL(path, req.url)`, which caused the `Location` response header to
contain `https://localhost:PORT/...`. The browser would then try to navigate
to `localhost`, which is unreachable from the user's machine.

## Fix

`getPublicOrigin()` reads the proxy-forwarded headers that carry the real
hostname:

1. **`X-Forwarded-Proto`** — preferred protocol (`https`)
2. **`X-Forwarded-Host`** — preferred hostname (`primordia.exe.xyz`)
3. **`Host`** — fallback (standard reverse-proxy behaviour preserves this)
4. `req.nextUrl.protocol` / `req.nextUrl.host` — last-resort fallback (the
   old behaviour, used in local dev where there is no proxy)

With the public origin correctly derived, the post-login redirect
(`/`) and the exe.dev login bounce (`/__exe.dev/login?redirect=...`) both
produce absolute URLs with the correct external hostname.
